# Контекст OpenCode: nkolinka.ru

Эта папка предназначена для создания и эксплуатации официального сайта АНО "Линка" на домене `https://nkolinka.ru`.

Перед любыми изменениями читай `README.md`, этот файл и документы в `docs/`. Общайся с пользователем на русском языке.

## Цель

- Новый сайт `nkolinka.ru` становится главным официальным сайтом АНО "Линка" и проекта Linka.
- Старый `linka.su` будет выключен позже, но редиректы на первом этапе не делаем.
- Первая версия переносит контент `linka.su` без комментариев. С 2026-07-08 разрешен отдельный раздел добровольных пожертвований.

## Принятые Решения

- GitHub repository: `linkasu/nko-linka.ru`.
- Yandex Cloud folder: `b1gn4stour811vgtjude`.
- Runtime: Yandex Cloud Serverless Containers.
- CMS: WordPress.
- Database: MySQL/MariaDB в Docker на сервере `37.230.192.57`.
- WordPress uploads: Yandex Object Storage.
- TLS: Yandex Certificate Manager.
- Secrets: GitHub Actions secrets для CI/CD, YC secrets/Lockbox для runtime.
- Admin user: только через secret, не хранить пароль в репозитории.
- Analytics: не подключать в первой версии, оставить на потом.
- Language: только русский.
- Design: стандартный сайт НКО, с символикой Линки.
- Contact form: не делать, использовать email.
- Legal address: полный адрес показывать в реквизитах и политике ПДн, не дублировать в каждом футере.
- Reports: создать пустой раздел отчетов с пояснением, что первые отчеты появятся после отчетного периода.
- Donations: разрешены только добровольные пожертвования на уставную деятельность АНО "Линка". До одобрения ЮKassa платежная кнопка должна быть неактивной заглушкой.

## Запреты

- Не добавлять оплату товаров, услуг, курсов, консультаций, программного обеспечения, цены, тарифы, корзину или коммерческие CTA без отдельного решения пользователя.
- Не включать активный прием пожертвований, платежную форму, платежную ссылку или платежные реквизиты до отдельного подтверждения после одобрения ЮKassa.
- Не сохранять секреты, токены, пароли, дампы БД, содержимое `wp-config.php`, private keys или значения production environment в репозиторий.
- Не собирать Docker images локально или на сервере. Сборка должна идти через GitHub CI и публиковаться в registry.
- Не запускать destructive-команды без явного подтверждения: `docker compose down`, `docker rm`, `docker volume rm`, `docker system prune`, `rm -rf`, DB drop/reset/import, массовые `wp search-replace`.
- Не менять старый `linka.su`, кроме отдельной явной задачи.
- Не публиковать ИНН физлица руководителя на HTML-страницах без отдельной необходимости; использовать публичные реквизиты юрлица.

## Рабочий Процесс

1. Сначала read-only инвентарь: git status, YC folder, сервер, DNS, текущие контейнеры.
2. Перед production-изменениями делать бэкап БД и критичных файлов.
3. Все sync/deploy операции сначала dry-run, затем apply только после проверки.
4. Минимизировать изменения и фиксировать решения в `docs/`.
5. После изменений проверять сайт, container revision, логи, HTTP status, WordPress health.
